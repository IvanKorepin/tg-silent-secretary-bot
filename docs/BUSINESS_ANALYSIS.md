## 1. Контекст и проблема

Telegram Business (доступен подписчикам Telegram Premium) позволяет подключить бота как «секретаря» — bot получает доступ к входящим сообщениям аккаунта и может отвечать от его имени. Стандартный сценарий использования — автоответчик.

**Проблема пользователя:** Подписчик множества каналов и чатов накапливает непрочитанные сообщения. Счётчик непрочитанных растёт, создавая «цифровой шум», хотя фактически содержимое каналов не требует реакции — достаточно их просто «обнулить».

**Решение:** Бот-секретарь автоматически проходит по выбранным чатам/каналам и отмечает сообщения как прочитанные — без привлечения внимания пользователя.

---

## 2. Цели продукта

| Цель | Метрика успеха |
|------|---------------|
| Снять «цифровой шум» от непрочитанных сообщений | 0 непрочитанных в выбранных каналах после прохода бота |
| Дать пользователю контроль над тем, что мониторится | Гибкий whitelist чатов/каналов |
| Минимальные данные — максимальная приватность | Никаких пользовательских данных не хранится на диске |

## 3. Аудитория

### Фаза 1 — MVP (текущая)
- **Один пользователь (автор)** — внутренний инструмент, self-hosted.

### Фаза 2 — B2C расширение
- **Telegram Premium пользователи** — единственная аудитория, у которой есть доступ к Telegram Business API.
- Портрет: активные потребители информации, подписанные на 50+ каналов, испытывающие «notification fatigue».

## 5. Нефункциональные требования

| ID | Требование |
|----|-----------|
| NF-01 | **Приватность:** никакие сообщения, сессионные данные и настройки не сохраняются на диск в MVP |
| NF-02 | **Стек:** Python 3.11+ + **aiogram 3.x** (Bot API враппер, нативная поддержка Business API) |
| NF-03 | **Надёжность:** при ошибке API не крашиться, логировать и продолжать |
| NF-04 | **Минимальный footprint:** бот не пишет в чаты, не отправляет уведомлений, работает «тихо» |
| NF-05 | **Telegram ToS:** не превышать rate limits Telegram API, не имитировать человека |

## 6. Архитектурные ограничения

### Telegram Business Bot — ключевые факты
- Бот подключается через **Telegram Business API** (не Bot API v1). Пользователь назначает бота секретарём в настройках TG Business.
- Бот получает апдейты `business_message` — новые входящие сообщения в разрешённых чатах.
- **Важно:** бот видит только те чаты, которые пользователь явно разрешил в настройках TG Business. Это ограничение платформы.
- Отметить сообщение как прочитанное можно методом `readBusinessMessage` (только через Business Bot connection).

### Что НЕ может бот в секретарском режиме
- Получать историю сообщений за прошлые периоды (только новые апдейты).
- Видеть чаты, не добавленные пользователем в TG Business UI.
- Отправлять сообщения от имени пользователя без явного разрешения.

## 7. Риски

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Telegram изменит Business API | Низкая | Высокое | Следить за changelog, тесты на методы |
| Превышение rate limits → бан аккаунта | Средняя | Критическое | Throttling между запросами, exponential backoff |
| Пользователь не имеет Telegram Premium | Высокая (для B2C) | Блокер | Явная проверка и информирование при onboarding |
| Потеря сессии при рестарте (in-memory) | Высокая | Средняя | Документировать, в Фазе 2 добавить persist сессии |

---

## 8. Зависимости

- **Telegram Premium / Business** — обязательная подписка у пользователя.
- **aiogram 3.x** — Bot API враппер с поддержкой Business-апдейтов и `readBusinessMessage`.
- **Python 3.11+** — язык разработки.
- **Anthropic API / Claude** (Фаза 2) — для генерации summary.
